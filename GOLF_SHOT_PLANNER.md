# Golf Shot Planner — Project Documentation

## Overview

Single-file vanilla HTML/CSS/JS golf shot planning web app. No dependencies, no build step. Hosted on GitHub Pages. Designed and tested for mobile (iPhone Safari + Brave). The entire application lives in `index.html`.

## Architecture

### File Structure

One HTML file, two `<script>` blocks:

| Block | Scope | Role |
|---|---|---|
| Script 1 | `DOMContentLoaded` closure | Core calculation engine, GPS, wind, carousel, club picker, course bar |
| Script 2 | `prepare-module` (`'use strict'`) | My Golf tab — stats, rounds, courses, score entry, profiles |

Functions needed by both blocks must be exposed via `window.*`. Safari: use explicit `window.x = y` assignments (not `defineProperty` on `window`).

### Two-Tab Layout

- **Play tab** (`#panePlay`) — strategy planning during a round
- **My Golf tab** (`#panePrepare`) — stats, history, bag, courses, profile

Tab switching: `window.switchTab('play')` / `window.switchTab('prepare')`

## Data Structures

### Course object (`golfCourses_v1[id]`)

```js
{
  name: string,
  courseRating: number,   // e.g. 72.1
  slopeRating: number,    // e.g. 125
  holes: [                // always 18 entries
    {
      par: 3 | 4 | 5,
      si: number,         // stroke index 1–18
      length: number,     // metres
      note: string        // optional hole note
    }
  ]
}
```

**Course IDs:** 5-digit numeric strings (10000–99999), generated by `newCourseId()`, collision-checked against existing courses. Older courses may have legacy `c_timestamp_random` IDs — both work correctly.

### Round object (`golfRounds_v1[courseId][]`)

```js
{
  date: string,           // 'YYYY-MM-DD'
  holesPlayed: number,
  totalStrokes: number,
  totalPar: number,
  totalPutts: number,
  totalGIR: number,
  scores: [               // 18 entries, null if not played
    {
      fairway: number,    // fairway shots taken
      rough: number,      // rough shots (0 or 1 flag, not added to score)
      putts: number,
      gir: boolean,       // green in regulation
      fir: boolean | null // fairway in regulation (null on par 3)
    }
  ],
  strategies: {           // keyed by hole index (number or string)
    [holeIdx]: string     // encoded: 'Max distance · Driver'
  }
}
```

### Score entry (in-progress, `roundScores_<courseId>`)

Array of 18 entries. Each entry: `null` or `{ fairway, rough, putts, gir, fir }`.

- `rough`: 0/1 flag — does NOT add to stroke count, but sets FIR=false and adds +0.35 rough penalty to expected strokes
- `gir`: `(fairway + rough) === (par - 2)`
- `fir`: `rough > 0` → always false; `fairway > 0` → true; par 3 → null

## Core Calculation Engine (Script 1)

### Club Data

Defined in the `clubs` array. Each club has:
- `key`: string identifier
- `label`: display name
- `relCarry`: fraction of driver carry (driver = 1.000)
- `rollSoft` / `rollFirm`: roll multiplier by conditions

Three user-entered anchors: driver carry, 7i carry, PW carry. All other carries interpolated via `interpolate()` in relCarry space.

### `getClubs(driver, i7, pw)`

Returns all checked clubs as objects: `{ key, carry, baseCarry, total, roll, idx }`.

- `carry` = wind+temperature adjusted carry (`applyWind(baseCarry, idx)`)
- `total` = `carry × roll`
- `baseCarry` = raw interpolated carry (stored for wind delta display)

### `findBestContinuation(teeClub, hole, driverTotal, clubsList, driverCarry)`

Finds the optimal continuation plan for a given tee club.

- **Single-shot:** if `hole - teeClub.total` lands within `maxApproach` (= `driverTotal × 0.92`), returns 1-shot plan
- **Two-shot:** iterates all non-driver candidates, picks lowest `expectedStrokesRemaining` score
- Always uses `driverCarry` (not teeClub.carry) as tier reference for `expectedStrokesRemaining`
- `maxApproach` uses wind-adjusted `driverTotal` (minor strategy set variation by conditions — acceptable)

### `expectedStrokesRemaining(approachDist, driverCarry, applyWindAdj?)`

Lookup table model:

```js
EXPECTED_STROKES = {
  bands:  [60, 100, 130, 160, 190, Infinity],  // approach distance bands
  tiers:  [195, 220, 250, Infinity],            // player ability tiers by driver carry
  values: [/* 6×4 matrix */]
}
```

Adjustments applied in order:
1. **HCP adjustment** — WHS per-hole model if course has CR/Slope/all 18 SIs; flat `hcp × 0.056` fallback
2. **Wind adjustment** — `±0.04 strokes/m/s` effective headwind, capped at −0.15/+0.25
3. **Rough penalty** — +0.35 strokes when `_inRough` is true

**Critical:** `_holeHcpAdj` and `_blCourseId`/`_blHoleIdx` must be computed **before** `ordered[]` is built in `calculate()`. This was a historical bug — these were previously set after the plans were scored.

### WHS Handicap Model

```
playingHcp = round(hcpIndex × Slope/113 + (CR − Par))
holeStrokes = floor(playingHcp/18) + (SI <= playingHcp%18 ? 1 : 0)
```

Requires: all 18 SI values set, CR > 0, Slope > 0. Falls back to flat model otherwise.

### `blendedScore(modelScore, courseId, holeIdx)`

Blends model score with personal baseline (average actual score on that hole).
- Requires ≥5 rounds of data for that hole
- Weight ramps 0→1 from 5→12 rounds
- Applied to `ordered[]` plans and inside `buildPlanWithShot2Override`

### Temperature & Wind

`tempCarryFactor()` reads `windState.tempC` (actual air temperature, NOT feels-like). Formula: air density ratio vs 15°C standard, scaled by 0.55 empirical factor. Effect: ≈±5m on 230m driver across Swedish season.

Applied only when wind is **enabled**. Disabling wind disables temperature correction too.

`applyWind(carry, clubIdx)`:
1. If wind disabled → return raw `carry`
2. Apply temperature correction: `carry × tempCarryFactor()`
3. If wind not active → return temp-adjusted carry
4. Apply headwind/tailwind using altitude-corrected wind speed

## Override System

Club overrides are namespaced by `courseId|holeIdx|strategyType` to prevent cross-hole/cross-course bleed.

```js
function _hk(type) { return _overrideCourseId + '|' + _overrideHoleIdx + '|' + type; }
```

`_overrideHoleIdx` and `_overrideCourseId` are:
- Initialised from sessionStorage at **script parse time** (IIFE)
- Re-synced at the start of every `calculate()` call

### Override objects

| Variable | Purpose |
|---|---|
| `teeOverrides` | User-selected tee club per strategy type |
| `shot2Overrides` | User-selected 2nd shot club per strategy type |
| `approachOverrides` | User-selected approach club (1-shot plans) |
| `gpsShot2Overrides` | Next club after GPS ball mark |
| `par3ClubOverrides` | Par 3 club, keyed by `courseId|holeIdx` |

Overrides cleared on: `calculate(true)` (Calculate button press). GPS overrides additionally cleared on `clearGpsState()` (hole navigation).

## Par 3 Handling

`calcPar3(clubsList, hole)`:
- Finds club with smallest positive `hole - c.total` (falls short, closest to pin)
- Falls back to club with smallest overshoot if nothing reaches
- Driver is excluded only if it grossly overshoots — user can still select any club via chip/bag picker

Score display: `remaining = Math.max(0, hole - s.total)` — overshooting clubs show negative distance in amber with "Overshoots — spin back" note. `_holeHcpAdj` and `blendedScore` both applied to par 3 scores (historically missed, fixed).

## Course Bar (`renderPlayCourseBar`)

Injected as first child of `#panePlay` when a course is loaded. Contains:
- Course name, hole number, par, distance
- Score vs par (current round)
- "Best here" strategy tag (from round history)
- Hole grid (2×9, front/back)
- GPS section
- Round summary (collapsible)

`updateBar()` rebuilds the header and grid on every hole navigation. `navigateTo(idx)` handles hole switching — also triggers `calculate()` via 50ms setTimeout.

### Best Strategy Tag

`renderLastRoundHint(idx)` — computes best-performing strategy for the current hole across all saved rounds, displays as: `Best here: [pill] N rounds`. No interaction, no expansion. Only shown when history exists.

## GPS System

Located inside `DOMContentLoaded` closure.

States: `idle` → tee marked → ball marked → reset.

- `gpsTeeSetState(state)` — controls tee tile visual state
- `gpsBallSetState(state, dist, shotN)` — controls ball tile
- `clearGpsState()` — resets all GPS state, clears `gpsShot2Overrides`

GPS tracking drives `completedShots[]` and `window.teeMarked`. These are read by `buildStrategyCard` to show GPS-adjusted expected strokes during a round.

## Wind System

`windState` object:
```js
{
  speedMs, gustMs, dirDeg, holeDeg,
  headwind, crosswind, active, enabled,
  tempC, feelsLike, rainPct
}
```

`computeWindComponents()` — derives headwind/crosswind from dirDeg vs holeDeg. Sets `windState.active = true` (never reset to false, only enabled/disabled).

Weather data from Open-Meteo (no API key). Fetched by `fetchWind()`.

**Display:** The "Temperature · carry" cell in the weather breakdown shows `tempC` (actual air temp) and carry delta. The weather strip header shows feels-like for comfort reference.

## UI Patterns

### Course Active State

When `#playCourseBar` exists in DOM:
- `#thisHoleCard` gets `.course-active` → manual hole fields hidden
- `#parCondGrid` (Par + Conditions strip) hidden entirely
- `#weatherCondRow` (Conditions chips in Weather section) shown
- Calculate button hidden; recalc link shown instead

### Conditions Chips

Two chip rows both drive the hidden `#conditions` select:
- `#condChipRow` — shown in Par/Conditions strip (no course loaded)
- `#condChipRowWeather` — shown below Weather header (course loaded)

Both wired via `wireChipRow()`. Synced on `conditions` change event.

### Chip Control Pattern

```js
wireChipRow('rowId', 'hiddenSelectId')  // wire clicks → select change
syncChipRow('rowId', 'hiddenSelectId')  // sync active class from select value
```

### Club Picker Modal

`window._openClubPicker(currentKey, onSelect, constraints)` — bottom sheet.

Constraints:
- `{ type: 'no-driver' }` — excludes driver
- `{ type: 'par', parValue }` — filters by par
- `{ type: 'window', defaultKey, clubsList, n }` — ±n clubs from default

Drag-to-dismiss: attaches to full sheet, activates in top 60px zone. Listeners cleaned up on close via `window._cpDragCleanup`.

## Critical Rules for AI Coding Assistants

1. **Always grep before replacing** — use Python `str.replace` with `assert s.count(old) == 1` before writing
2. **Validate syntax after every change** — `node --check` on the prepare-module script block
3. **Check div balance** when touching HTML — especially the wind panel (3 levels deep: `windSection > collapsible-header + collapsible-body > compassSection`)
4. **`_holeHcpAdj` must be set BEFORE `ordered[]` is built** in `calculate()`
5. **`driverCarry` (not `teeClub.carry`)** must be passed to `expectedStrokesRemaining` everywhere
6. **Override keys use `_hk(type)`** — never bare strategy type strings
7. **Safari compatibility** — no `Object.defineProperty` on `window`; use explicit assignments
8. **`blendedScore` must be applied** in `buildPlanWithShot2Override` AND in `buildDeltaSection` AND in `getActivePlan` inside `buildCompareTable`
9. **Copyright comment** must be at the top of every output file
10. **Test on both Brave and Safari** — subtle JS differences have caused cross-browser bugs throughout
